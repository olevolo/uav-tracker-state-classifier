## What This Project Is

A UAV object tracker that routes each frame to the appropriate tracker tier using a **learned scene classifier**, not a hand-tuned entropy threshold. The original `uav-entropy-tracker` repo used Shannon entropy of optical flow as the switching signal; this repo replaces that with a 6-class neural scene classifier (`MobileNetV3-Small` backbone) trained on UAV123 difficulty labels.

**This repo is the v2-ml-extension branch of `../uav-entropy-tracker`, cleaned of v1 entropy components and placed in its own repo.**

## Architecture in One Picture

```
Per-frame pipeline (target: 33ms / 30fps):

  Frame
    │
    ├─► SceneClassifier (CNN, every 5 frames, <3ms)
    │       ↓ SceneClass: CLEAR/MODERATE/CHALLENGING/RISK_LOSS/RECOVERY/LOW_RES
    │
    ├─► MotionEntropySignal (fallback when classifier uncertain, <2ms)
    │
    ├─► MLSceneScheduler → tier decision (scene class → tier mapping)
    │       ↓ falls back to MultiTierScheduler when confidence < 0.60
    │
    ├─► Active tracker (by tier):
    │       Tier 0: KCF Henriques (~0.04 GFLOPs, ~5ms CPU)
    │       Tier 1: [reserved — MobileTrack removed, slot open]
    │       Tier 2: OSTrack-256 (~4.8 GFLOPs stub, ~6ms CPU stub)
    │       Tier 3: YOLOv8n detector (~8.7 GFLOPs, recovery only)
    │
    ├─► AppearanceMemory.store() (if confidence >= 0.6)
    ├─► MotionPredictor.update() (online SGD step)
    └─► TelemetryEntry (bbox, confidence, tier, scene_class, motion_pred_bbox, ...)
```

## Training Pipeline (to get real accuracy numbers)

```bash
# Step 1: Generate per-frame difficulty labels (runs KCF on all 123 UAV123 sequences)
# Takes ~20-30 min CPU, 4 workers, outputs ~50MB NPZ
python scripts/generate_ml_labels.py \
  --dataset-root $UAV_DATA_ROOT/uav123 \
  --output $UAV_DATA_ROOT/uav123_labels.npz \
  --tracker kcf_henriques \
  --seed 42

# Step 2: Install timm for real MobileNetV3-Small backbone
pip install timm>=0.9.0

# Step 3: Train scene classifier (30 epochs, ~2hrs GPU / ~12hrs CPU)
python scripts/train_scene_classifier.py \
  --config configs/training/scene_classifier_uav123.yaml \
  --seed 42
# Output: weights/scene_classifier/best_val_f1.pth + ONNX export

# Step 4: Download OSTrack weights
# From: https://github.com/botaoye/OSTrack
# Place at: weights/ostrack/ostrack256_full_ep300.pth.tar

# Step 5: Evaluate full v2 pipeline
uav-tracker evaluate \
  --config configs/experiments/v2_full_ml.yaml \
  --dataset uav123 \
  --limit 50
```

## Key Design Decisions

All in `docs/adr/`:

| ADR | Decision |
|-----|---------|
| 0009 | **6 scene classes**: CLEAR / MODERATE / CHALLENGING / RISK_LOSS / RECOVERY / LOW_RES. Priority order for label assignment. UAV123 attribute mapping. |
| 0010 | **5 ML Protocols**: SceneClassifier, DifficultyPredictor, AppearanceMemory, MotionPredictor, ModelWarmer — all `@runtime_checkable`. 5 new Registry instances. |
| 0011 | **Train/val/test split**: alphabetical 98/12/8 by sequence name. Fixed constants in `UAV123MLDataset`. No stratification (too few sequences). |
| 0012 | **Label generation**: KCF IoU simulation → priority rules. RECOVERY > RISK_LOSS > CHALLENGING > LOW_RES > MODERATE > CLEAR. |
| 0013 | **OSTrack-256 as default tier-2**: faster than STARK-S50 (4.8 vs 6.2 GFLOPs). Both registered. Stub ConvNet validates pipeline before weights loaded. |
| 0014 | **ModelWarmer contract**: all models warmed before frame 0. 3 dummy passes. `is_warmed` property. HybridRunner calls `warmer.warmup()` at run start. |
| 0015 | **Online adaptation bounds**: backbone frozen in SceneClassifierOnlineAdaptor. Max 50 appearance templates. LSTM SGD disabled on `status=="lost"`. |

## How HybridRunner Uses ML Modules

```python
# Construct the full v2 pipeline:
from uav_tracker.runner import HybridRunner
from uav_tracker.ml.scene_classifier.cnn_classifier import MobileNetV3TinyClassifier
from uav_tracker.ml.appearance_memory.cosine_memory import CosineAppearanceMemory
from uav_tracker.ml.motion_predictor.lstm_predictor import OnlineLSTMMotionPredictor
from uav_tracker.ml.warmer.model_warmer import DefaultModelWarmer
from uav_tracker.schedulers.ml_scene_scheduler import MLSceneScheduler
from uav_tracker.registry import TRACKERS

clf = MobileNetV3TinyClassifier(device="cpu")

runner = HybridRunner(
    trackers={
        0: TRACKERS.build("kcf_henriques"),
        2: TRACKERS.build("ostrack_256"),
    },
    signals={"motion_entropy": MotionEntropySignal()},
    scheduler=MLSceneScheduler(scene_classifier=clf),
    scene_classifier=clf,              # wires classify() into frame loop
    appearance_memory=CosineAppearanceMemory(),
    motion_predictor=OnlineLSTMMotionPredictor(),
    warmer=DefaultModelWarmer(target_latency_ms=5000.0),
    seed=42,
)

for entry in runner.run(sequence):
    # entry.tier       ← 0 (KCF) or 2 (OSTrack)
    # entry.aux["scene_class"]        ← SceneClass int (0-5)
    # entry.aux["scene_confidence"]   ← float
    # entry.aux["motion_pred_bbox"]   ← (x, y, w, h) tuple from LSTM
    pass
```

## MobileTrack Data (Important Context)

The real MobileTrack results (IET 2022 paper) are pre-computed and available at:
```
~/projects/MobileTrack/                          ← full MobileTrack codebase
~/projects/uav-entropy-tracker/mobiletrack_data/ ← pre-computed UAV123 tracking results
  └── [IET 2022] MobileTrack/Results/UAV123/
      └── checkpoint_e15_wi-0.061_pk-0.130_lr-0.782/
          ├── bike1.txt, bike2.txt, ...           ← per-sequence bbox predictions
```

These are `.txt` files with one `x,y,w,h` bbox per line. They can be loaded directly for evaluation without running the model — useful for benchmarking what a real trained MobileTrack achieves before loading weights into this codebase.
